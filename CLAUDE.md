# COURTSIDE 개발 룰

## 절대 원칙

1. **요청하지 않은 변경 금지**: 요청한 작업만 정확히 수행한다. 기존 UI, 레이아웃, 로직을 임의로 변경하지 않는다. 임의 변경은 로직 꼬임의 원인이 된다.

2. **문서 선행 업데이트**: 모든 변경/수정사항은 프로토타입 작업 전에 관련 문서(요구사항, 화면설계서, 비즈니스 로직 등)를 먼저 업데이트한다.

3. **push 전 최신 동기화**: GitHub API로 push 하기 전 반드시 `git fetch origin main && git reset --hard origin/main`으로 최신 상태를 반영한 후 작업한다. 다른 세션의 커밋을 덮어쓰지 않는다.
   - push 직전에 GitHub API에서 해당 파일의 최신 내용을 다운로드하여 그 위에 수정한다. 로컬에 캐시된 오래된 파일을 push하지 않는다.
   - 세션 시작 시 `git log`로 마지막 커밋 확인. 다른 세션(데스크탑 등) 커밋이 있으면 반드시 전체 반영 후 작업 시작한다.
   - 데스크탑과 웹 세션에서 동일 파일을 동시에 작업하지 않는다.

4. **기존 디자인 존중**: 버튼 배치, 컬러, 레이아웃 등 기존에 확정된 디자인은 명시적 요청 없이 변경하지 않는다. 추가 요청 시 기존 구조를 유지하면서 추가만 한다.

5. **완료 전 자체 테스트 필수**: 작업 완료 보고 전에 반드시 Playwright 등으로 스크린샷을 찍어 인터랙션이 정상 동작하는지 직접 확인한다. HTML/JS 추가 시 함수 존재 여부, 바텀시트 노출, 버튼 클릭 동작을 검증한 후에만 완료 보고한다. 테스트 실패 시 수정 후 재테스트한다.

6. **화면 보고 시 링크 필수**: 화면 스크린샷을 보여줄 때 반드시 GitHub Pages 링크를 함께 제공한다. 사용자가 직접 모바일에서 확인할 수 있도록 한다.

### 규칙

- 기능 개발 시 해당 FR 번호와 screen-spec의 화면 번호를 먼저 확인한다.
- DB 스키마에 정의된 테이블/필드를 기준으로 구현하고, 스키마에 없는 필드를 임의로 추가하지 않는다.
- 비즈니스 로직 문서에 정의된 정책(회차 계산, 결제 상태, 변경 요청 플로우 등)을 그대로 따른다.
- 디자인 가이드의 컬러/타이포/컴포넌트 스펙을 준수한다.
- 프로토타입 HTML의 인터랙션을 참고하되, 스펙 문서가 우선한다 (충돌 시 문서 기준).

### 완료된 기능 (7-0 코치 스케줄 홈)
1. 주간 미니 캘린더 (날짜 클릭 전환 + ? 툴팁 범례)
2. 오늘 레슨 목록 — 상태별 카드: 진행중/레슨예정/변경요청/레슨완료/결강/보강 일정 선택중/보강확정
3. 상태 필터 칩 (전체/진행중/예정/변경/완료/결강/보강 + 건수 표시 + 다중 선택)
4. 레슨 상세 풀팝업 — 8개 상태별 UI + 회차/결제상태/결제일 표시
5. 상태처리하기 버튼 (전체 상태에서 노출) → 바텀시트: 레슨완료/보강처리/결강처리/결제확인(미결제시만)
6. 보강처리: 사유선택(고객/코치) + 상세입력 + 주단위 날짜 선택
7. 결강처리: 사유선택 + 상세입력 (결강 상태에서 결강처리 비노출)
8. 결제확인: 결제수단 선택(현금/계좌이체/기타) → 외부결제 전환
9. 변경요청 상세: 기존일정(취소선) + 희망날짜(확정하기 버튼) + 추천슬롯(다중선택) + 선택/모두 제안하기
10. 대기신청 상세: 스케줄 확정하기 + 더보기(보류/거절)
11. 변경요청 탭: 4종 카드(제안됨/변경가능/변경불가/변경완료)
12. 상태 흐름도 페이지 (표/차트 모드 전환 + 아코디언)
13. LNB 사이트맵에 작업현황표/상태흐름도 링크
14. 코치 전용 5탭 하단 네비바 — 9개 화면 통일 (7-0~7-9, 7-10)
15. 수강생 레슨 상세 (6-6): 결제상태 뱃지 URL 파라미터 연동 (payment, depositAmount, totalAmount)
16. 예약금 수령 (DEPOSIT_RECEIVED): 코치가 금액 입력 → 수강생 화면에 예약금/잔액 분리 표시
17. 잔액 수령 확인: 코치 레슨 상세 내 [잔액 수령 확인] 버튼 → PAID 전환
18. 회원 검색 (7-10): 실시간 검색 + 4종 정렬 (기본순/이름순/최근레슨순/미결제우선) + 수강생 카드

### 진행 예정

- 결강처리 바텀시트 UI 개선 (보강처리와 동일한 패턴으로)
- 프론트엔드/백엔드 개발 (Sprint 계획은 이전 세션에서 정의)
