### 소셜 로그인 응답 분기 (핵심 로직)

```
소셜 로그인 버튼 클릭
  │
  ├── OAuth 인증 실패
  │     └→ 토스트: "로그인에 실패했습니다. 다시 시도해주세요"
  │
  └── OAuth 인증 성공 (socialId 수신)
        │
        └── POST /api/auth/login/{provider} 호출
              │
              ├── Case 1: 기존 회원 (isNewUser=false)
              │     ├── isActive=true  → JWT 발급 → **2-1 홈**
              │     └── isActive=false → 토스트: "이용이 제한된 계정입니다"
              │
              ├── Case 2: 미가입 (isNewUser=true)
              │     └→ 바텀시트 표시 (아래 상세)
              │
              └── Case 3: 네트워크/서버 오류
                    └→ 토스트: "네트워크 연결을 확인해주세요"
```

### 미가입 바텀시트 상세 (Case 2)

소셜 인증은 이미 완료된 상태이므로, 회원가입 화면에서 소셜 인증을 다시 하지 않도록 **약관 동의만 받고 가입 진행**합니다.

| 요소 | 동작 | 로직 | 이동 |
|------|------|------|------|
| **바텀시트 헤더** | 표시 | 소셜 프로필 정보 표시 (프로필 사진, 이름, 이메일) + "가입된 계정이 없습니다. 회원가입을 진행할까요?" | |
| 전체 동의 | 체크 | 하위 3개 약관 모두 체크/해제 | |
| [필수] 이용약관 | 체크 | 개별 토글 | |
| [필수] 개인정보 처리방침 | 체크 | 개별 토글 | |
| [선택] 마케팅 수신 | 체크 | 개별 토글 | |
| 회원가입 진행 | 클릭 | ① 필수 약관 동의 확인 → ② `POST /api/auth/signup` (소셜 데이터 + 약관 동의 전달) → ③ JWT 발급 | → **1-3 역할선택** |
| 취소 | 클릭 | 바텀시트 닫기, 로그인 화면 유지 | |

**유효성 검사**:
- 필수 약관 미동의 → "회원가입 진행" 버튼 비활성화

**장점 (기존 대비 개선)**:
- 소셜 인증을 2번 하지 않음 (로그인 화면에서 이미 OAuth 완료)
- 사용자 이탈 최소화 (토스트 에러 → 별도 회원가입 화면 이동 대신, 바텀시트에서 즉시 가입)
- 소셜 프로필 정보 미리 표시로 신뢰감 제공

**API 응답 스펙**:
```json
// POST /api/auth/login/{provider}
// Case 1: 기존 회원
{ "isNewUser": false, "token": "jwt...", "user": { ... } }

// Case 2: 미가입
{ "isNewUser": true, "socialProfile": { "name": "홍길동", "email": "hong@email.com", "profileImage": "url...", "provider": "KAKAO", "socialId": "12345" } }
```

**에러 처리**:
- 소셜 OAuth 실패 → 토스트: "로그인에 실패했습니다. 다시 시도해주세요"
- 차단된 계정 → 토스트: "이용이 제한된 계정입니다. 고객센터에 문의하세요"
- 회원가입 진행 중 서버 오류 → 토스트: "회원가입에 실패했습니다. 다시 시도해주세요"
- 네트워크 오류 → 토스트: "네트워크 연결을 확인해주세요"

## 약관 상세 페이지

- **파일**: `flow1-onboarding/terms-detail.html?code={TERMS_CODE}`
- **FR**: FR-14f
- **진입 경로**: 학생 프로필(1-4) / 코치 프로필(1-6) → 약관 "보기 ›" 클릭

| 요소 | 동작 | 로직 | 이동 |
|------|------|------|------|
| ← 뒤로가기 | 클릭 | `history.back()` — 이전 화면(프로필 등록) 복귀 | → 이전 화면 |
| 약관 본문 | 페이지 로드 | URL 파라미터 `?code=` 추출 → `GET /api/terms/:code` [공개 API] → Terms.content(HTML) 렌더링 | |
| 확인 버튼 | 클릭 | `history.back()` — 이전 화면 복귀 | → 이전 화면 |

**URL 파라미터별 약관**:
| code | 약관명 |
|------|--------|
| `TERMS_OF_SERVICE` | 이용약관 |
| `PRIVACY_POLICY` | 개인정보 처리방침 |
| `MARKETING` | 마케팅 정보 수신 동의 |

**표시 데이터**: 제목(title), 버전(version), 시행일(effectiveDate), 본문(content — 어드민에서 입력한 HTML)

**API 호출**:
- `GET /api/terms/:code` [공개] — code별 isActive=true인 최신 버전 반환

**데이터 소스**: Terms 테이블 (어드민 FR-14g에서 CRUD 관리)

| **[변경 가능 카드]** | 표시 | `GET /api/bookings?role=coach&rescheduleStatus=PENDING` — 자동 추천 결과 슬롯 ≥ 1개인 건. 초록 보더 + "변경 가능" 배지. 카드 구성: 수강생명 + 레슨 정보 + **일정 통합 박스** (주황 배경 1개 박스에 기존일정 취소선 → 희망날짜 표시, 희망 날짜 가용 시 우측 확정하기 버튼 코랄색 강조) + **추천 슬롯 세로 리스트** (체크박스 + 날짜, 희망 날짜와 동일 슬롯은 제거하여 중복 방지, 타이틀 "변경 가능한 다른 날짜") | |
| 모두 제안하기 | 클릭 | ① `POST /api/bookings/:id/reschedule/propose { slotIds }` → ② ScheduleHold 생성 (12시간 홀드) → ③ rescheduleStatus = PROPOSED → ④ 수강생 알림톡 (제안 날짜 목록) → ⑤ 성공 토스트 "일정이 제안되었습니다" + 카드 상태 "제안됨"으로 변경 | |
| **[변경 불가 카드]** | 표시 | 자동 추천 결과 슬롯 = 0개인 건 (코치 가용 시간 없음 또는 다른 홀드로 슬롯 소진). 빨간 보더 + "변경 불가" 배지. 사유 메시지 표시 | |
| 불가 알림 보내기 | 클릭 | ① 확인 모달 ("기존 일정으로 진행됩니다") → ② `PUT /api/bookings/:id/reschedule/reject { reason: "UNAVAILABLE" }` → ③ rescheduleStatus = REJECTED → ④ 수강생 알림 (기존 일정 유지) | |
| **[제안됨 카드]** | 표시 | rescheduleStatus = PROPOSED인 건. 회색 보더 + "제안됨 · 응답 대기 중" 배지. 홀드 남은 시간 표시 (예: "남은 시간 8:32:15"). 수강생 응답 대기 상태 | |
| **[변경 완료 카드]** | 표시 | rescheduleStatus = COMPLETED인 건 (당일까지만 노출). 파란 보더 + "변경 완료" 배지. **일정 통합 박스** (파란 배경 1개 박스에 변경 전 취소선 → 변경 후 강조 + "확정" 배지). 다음날 00:00 이후 목록에서 자동 제거 | |

**코치 전용 하단 네비게이션** (공통 플로팅 라운딩 스타일 적용):

| 탭 | 아이콘 | 이동 |
|---|---|---|
| 📅 스케줄 (활성) | 현재 화면 | — |
| 💬 메시지 | 채팅 목록 | → **4-1 채팅목록** |
| 🔍 검색 | 코치 목록 | → **2-2 코치목록** |
| 👤 마이 | 코치 마이 | → **7-1 코치마이** |

**표시 데이터**:
- 오늘 레슨: `GET /api/bookings?role=coach&date=today&status=CONFIRMED,COMPLETED` [인증: 코치] — 확정(CONFIRMED) 건은 "레슨 예정"으로 표시, COMPLETED 건은 "레슨완료"로 표시. 오늘 레슨 목록에서 "확정" 배지는 노출하지 않음. **레슨 시간은 시작 시간만 표시** (종료 시간 미노출)
- 변경 요청: `GET /api/bookings?role=coach&rescheduleStatus=PENDING,PROPOSED,COMPLETED&sort=rescheduleRequestedAt:ASC` [인증: 코치] — 먼저 요청한 수강생 순 정렬 (선입선출). 각 건에 자동 추천 결과 포함. COMPLETED는 당일까지만 반환
- 대기 신청: `GET /api/bookings?role=coach&status=PENDING&limit=3` [인증: 코치]
- 오늘 레슨 카드 표시 정보: 시작 시간 + 수강생명 + 레슨 형태(1:1/그룹) + **현재 회차(N/M회)** + 결제 상태 + 상태 배지. 레슨 목적은 미표시
- 오늘 레슨 상태 필터: 필터 아이콘 1개 클릭 → 바텀시트에서 상태 다중 선택 (진행중/예정/변경/완료/결강/보강). 적용 시 선택한 상태만 표시. 상태 범례는 "?" 툴팁으로 표시
- 이번 주 요약: 총 레슨 수 / 대기 신청 수 / 완료 수
- 홀드 상태: `GET /api/schedule-holds?coachId=me` [인증: 코치] — 현재 활성 홀드 목록 (남은 시간 계산용)
