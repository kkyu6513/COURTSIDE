## 1. 홈 화면 섹션 노출 로직

### 1.1 우리동네 코치님

**4가지 케이스 분기:**

| 케이스 | 조건 | 노출 | 이동 |
|--------|------|------|------|
| A. 비로그인 | JWT 없음 | 로그인 유도 안내 | → 로그인 |
| B. 지역 미설정 | 로그인 + preferredAreaSido = null | 지역 설정 유도 | → 테니스 정보 |
| C. 코치 있음 | 로그인 + 지역 설정 + 해당 지역 코치 1명+ | 코치 카드 목록 | → 코치 상세/목록 |
| D. 코치 없음 | 로그인 + 지역 설정 + 해당 지역 코치 0명 | 빈 상태 안내 | → 코치 목록 |

**조회 로직 (Case C):**
```
API: GET /api/coaches?sido={preferredAreaSido}&sigungu={preferredAreaSigungu}&limit=5&sort=rating
인증: 필요 (수강생/코치)

정렬 우선순위:
  1차: averageRating DESC (평점 높은 순)
  2차: reviewCount DESC (리뷰 많은 순)
  3차: createdAt DESC (최신 등록순)

확대 로직:
  해당 시·군·구에 5명 미만 → 동일 시·도의 다른 시·군·구 코치로 확대 조회
  확대 시 원래 시·군·구 코치를 우선 배치
```

### 홈 화면 코치 카드 공통 구성 (우리동네/핫한/신규 동일)

| 요소 | 데이터 경로 | 표시 |
|------|-----------|------|
| 프로필 사진 | `User.profileImage` | 원형 52px. 미등록 시 기본 그라디언트 |
| 코치명 | `User.name` | 14px Bold |
| 지역 (1뎁스 + 2뎁스) | `CoachProfile.areaSido` + `CoachProfile.areaSigungu` | "서울특별시 강남구" (전체 노출) |
| 성별 | `CoachProfile.gender` | MALE → "남성" / FEMALE → "여성" |

**미표시 항목** (카드에서 제외):
- ~~전문분야~~ — 코치 상세에서 확인
- ~~평점 (★)~~ — 코치 상세에서 확인
- ~~구독 배지 (PRO/ANNUAL/FREE)~~ — 코치 상세에서 확인

### 1.2 지금 핫한 코치님

**노출 조건:** 로그인 여부 무관 (공개 API)

**인기 점수 계산:**
```
popularityScore = (최근 30일 bookingCount) × 0.4
               + averageRating × 0.3
               + reviewCount × 0.3
```

**최소 조건:**
- `reviewCount >= 3` (리뷰 3개 이상)
- `averageRating >= 4.0` (평점 4.0 이상)
- 조건 미충족 코치는 목록에서 제외

**조회 로직:**
```
API: GET /api/coaches?sort=popular&limit=5 [공개]

정렬: popularityScore DESC
노출: 최대 5명 (가로 스크롤)
갱신: 서버 캐시 1시간 (배치 계산, 실시간 아님)
빈 상태: 조건 충족 코치 0명이면 섹션 숨김 (서비스 초기)
```

**HOT 배지:**
- 인기 점수 상위 3명에게만 🔥 HOT 배지 표시
- 4~5위는 배지 없이 일반 카드

### 1.3 신규등록 코치님

**목적:** 신규 코치에게 초기 노출 기회를 제공하여 첫 레슨 신청 유도 + 코치 유입 촉진

**노출 조건:** 로그인 여부 무관 (공개 API)

**조회 로직:**
```
API: GET /api/coaches?sort=newest&limit=5 [공개]

정렬: CoachProfile.createdAt DESC (등록일 최신순)
필터:
  - createdAt >= NOW() - 30일 (최근 30일 이내 등록)
  - 프로필 필수 항목 모두 완료 (이름, 소개, 전문분야 1개+, 지역, 소개 이미지 1장+)
  - User.isActive = true (차단 코치 제외)
  - CoachProfile.isVerified는 무관 (인증 없어도 노출)
노출: 최대 5명 (가로 스크롤)
갱신: 서버 캐시 30분
빈 상태: 조건 충족 코치 0명이면 섹션 자체 숨김
```

**30일 경과 후 처리:**
```
코치 등록일이 30일을 초과하면:
  → 신규등록 섹션에서 자동 제외 (쿼리 조건으로 자동, 별도 처리 없음)
  → 해당 코치는 "핫한 코치님"이나 일반 검색 결과에서 노출
  → NEW 배지도 자동 제거 (프론트에서 createdAt 기준 30일 체크)
```

**코치 카드 구성:** 홈 화면 공통 카드 구성 참조 (프로필 사진 + 코치명 + 지역 전체 + 성별)

**"핫한 코치님"과의 차이:**

| | 핫한 코치님 | 신규등록 코치님 |
|---|-----------|-------------|
| 정렬 | 인기 점수 (가중치 계산) | 등록일 최신순 |
| 최소 조건 | 리뷰 3개+ & 평점 4.0+ | 프로필 완료만 |
| 기간 필터 | 없음 (전체 기간) | 최근 30일 이내 |
| 목적 | 검증된 인기 코치 노출 | 신규 코치 초기 노출 기회 |

## 2. 코치 검색/필터 로직

### 2.1 검색 조건

```
API: GET /api/coaches [공개]

파라미터:
  keyword        (String)  — 코치명 부분 일치 (LIKE %keyword%)
  sido           (String)  — 시·도 정확 일치
  sigungu        (String)  — 시·군·구 정확 일치
  gender         (Enum)    — MALE / FEMALE / 미지정(전체)
  ntrp           (String)  — NTRP 범위 필터 (ntrpMin <= 값 <= ntrpMax)
  priceMin       (Int)     — 최소 가격
  priceMax       (Int)     — 최대 가격
  lessonFormat   (Enum)    — INDIVIDUAL / GROUP / 미지정(전체)
  experienceMin  (Int)     — 최소 경력 (년)
  experienceMax  (Int)     — 최대 경력 (년)
  sort           (String)  — rating(기본) / popular / newest / price_asc / price_desc
  page           (Int)     — 페이지 번호 (기본 1)
  limit          (Int)     — 페이지당 건수 (기본 20)
```

### 2.1a 필터 바텀시트 UI 구성

각 필터 칩 클릭 → 바텀시트 열기 → 옵션 선택 → "적용하기" → 목록 재조회

| 필터 | 바텀시트 타입 | 옵션 |
|------|-------------|------|
| 지역 | 셀렉트 2개 (시·도 + 시·군·구 연동) | 전체 + 17개 시·도 → 시·군·구 |
| 성별 | 칩 단일 선택 | 전체 / 남성 / 여성 |
| 경력 | 칩 단일 선택 | 전체 / 1~3년 / 3~5년 / 5~10년 / 10년+ |

※ NTRP, 가격대, 정렬, 레슨유형 필터는 MVP에서 제외 (코치 상세에서 확인)

### 2.2 정렬 옵션

| sort 값 | 정렬 기준 |
|---------|----------|
| `rating` | averageRating DESC, reviewCount DESC |
| `popular` | popularityScore DESC (1.2 인기 점수) |
| `newest` | createdAt DESC |
| `price_asc` | lessonPrice ASC (비공개 제외) |
| `price_desc` | lessonPrice DESC (비공개 제외) |

### 2.3 가격 비공개 코치 처리

- `priceVisibility = PRIVATE` 코치: 가격 필터 적용 시 결과에서 제외
- 가격 필터 미사용 시: 비공개 코치도 포함 (가격 대신 "가격 문의" 표시)

## 3. 구독 플랜 정책

### 3.1 플랜별 기능 제한

| 기능 | 무료 | 월간 PRO | 연간 PRO |
|------|:----:|:--------:|:--------:|
| 프로필 기본 노출 | ✓ | ✓ | ✓ |
| 신청 수신 | 월 5건 | 무제한 | 무제한 |
| 검색 상위 노출 | ✗ | ✓ | ✓ |
| 통계 대시보드 | ✗ | ✓ | ✓ |
| PRO/ANNUAL 배지 | ✗ | PRO | ANNUAL |
| 인증 배지 우선 심사 | ✗ | ✗ | ✓ |
| 스케줄 등록 | ✗ | ✓ | ✓ |

### 3.1a 구독 배지 노출 정책

```
PRO / ANNUAL / FREE 구독 배지:
  → 코치 상세 페이지에서 관리자(User.role = ADMIN)에게만 노출
  → 일반 사용자(수강생/코치/비로그인)에게는 미표시
  → 목적: 구독 플랜은 코치 내부 관리 정보이므로 수강생에게 공개하지 않음
  → 인증 배지(✓ 인증)는 모든 사용자에게 공개 (관리자 심사 완료 표시)
```

### 3.2 검색 상위 노출 로직

```
코치 목록 정렬 시:
  1단계: 구독 플랜 우선순위 — YEARLY(1) > MONTHLY(2) > FREE(3)
  2단계: 동일 플랜 내에서 선택된 sort 기준 적용

예: sort=rating일 때
  [YEARLY 코치들 (rating순)] → [MONTHLY 코치들 (rating순)] → [FREE 코치들 (rating순)]
```

### 3.3 무료 플랜 신청 수신 한도

```
월 5건 한도 체크:
  SELECT COUNT(*) FROM Booking
  WHERE coachId = {coachId}
    AND status = 'PENDING'
    AND createdAt >= DATE_TRUNC('month', NOW())

  결과 >= 5 → 신규 신청 거부 + 수강생에게 "해당 코치의 이번 달 신청 한도가 가득 찼습니다" 안내
```

### 3.4 구독 자동 갱신

```
스케줄러: 매일 00:00 실행

1. endDate가 오늘인 CoachSubscription 조회
2. autoRenew = true:
   → 토스페이먼츠 자동 결제 요청
   → 성공: endDate = endDate + 1개월(월간) 또는 1년(연간)
   → 실패: status = EXPIRED, 코치에게 결제 실패 알림, 3일 유예
3. autoRenew = false:
   → status = EXPIRED
   → 코치 기능을 무료 플랜으로 자동 전환
```

### 3.5 구독 해지 정책

```
해지 요청 시:
  autoRenew = false로 변경
  status는 ACTIVE 유지 (endDate까지 기능 사용 가능)
  endDate 이후 → status = EXPIRED → 무료 플랜으로 전환
```

## 4. 레슨 신청/예약 로직

### 4.1 예약 상태 머신 (신청 방식별 분기)

```
[레슨 신청만 (미결제)]
  POST /api/bookings (status=PENDING, paymentId=null)
  
  PENDING (대기)
    ├── 코치 수락 (스케줄 확정) → CONFIRMED (확정, 미결제)
    ├── 코치 보류 → ON_HOLD (보류, holdReason 필수 → 수강생 알림톡 발송)
    ├── 코치 거절 → REJECTED (거절, 사유 필수)
    └── 수강생/코치 취소 → CANCELLED (취소, 사유 필수)
  
  ON_HOLD (보류)
    ├── 코치 수락 (스케줄 확정) → CONFIRMED (확정)
    ├── 코치 거절 → REJECTED (거절, 사유 필수)
    └── 수강생/코치 취소 → CANCELLED
  
  CONFIRMED (확정, 미결제)
    ├── 레슨 완료 → COMPLETED (완료)
    └── 취소 → CANCELLED
  
  ※ 미결제 CONFIRMED: 스케줄 관리 기능 이용 불가

[결제 후 신청]
  POST /api/payments → POST /api/bookings (status=CONFIRMED, paymentId=결제ID)
  
  CONFIRMED (확정, 결제 완료)
    ├── 스케줄 자동 블록 (즉시)
    ├── 수강생 스케줄 관리 기능 활성화
    ├── 레슨 완료 → COMPLETED (완료)
    └── 취소 → CANCELLED (환불 처리)

COMPLETED (완료)

[보류 처리]
  PUT /api/bookings/:id/hold { reason } [인증: 코치]
    → Booking.status = ON_HOLD
    → Booking.holdReason = reason (필수)
    → Booking.holdAt = NOW()
    → 수강생에게 BOOKING_ON_HOLD 알림톡 발송 (보류 사유 포함)
    → 보류 중 신청은 "대기 중 신청" 목록에 유지

[대체 일정 제안 — 슬롯 충돌 시]
  대기 신청에서 수강생 희망 일정 중 이미 블록된 슬롯이 있는 경우:
    → 코치 레슨 상세에 레슨가능/레슨불가 구분 표시
    → 레슨불가 슬롯에 대해 코치가 대체 날짜를 선택하여 제안
    → 대체 날짜 UI: 주단위/일단위 탐색 (‹ 이전주/다음주 ›)
      날짜별 가용 슬롯 표시 (체크박스 다중 선택)
      추천 슬롯 상단 우선 표시: 1순위 같은 요일 ±1시간, 2순위 ±1일 같은 시간, 3순위 ±1일 다른 시간
      추천 슬롯에는 "추천" 배지 표시

  코치 제안 시:
    POST /api/bookings/:id/propose { availableSlotIds, proposedSlotIds }
    → 레슨가능 건: 즉시 스케줄 블록 (확정 대기)
    → 제안 건: 수강생에게 BOOKING_PROPOSAL 알림톡 발송
    → 수강생이 제안 날짜 중 1개 선택 → PUT /api/bookings/:id/accept-proposal { selectedSlotId }
    → 전체 CONFIRMED + 코치 알림 발송
    → 수강생 미응답 시 제안 만료 (홀드 정책 적용)

  수강생 선택 화면:
    → 알림톡 "일정 선택하기" → 6-6 신청 상세 (제안 선택 모드)
    → 확정 일정 (읽기 전용) + 제안 일정 (라디오 선택) + 선택 완료 버튼

[미결제 신청 미확정 리마인더]
  수강생이 미결제(isPaid=false)로 레슨 신청 후, 코치가 스케줄 확정하지 않은 경우:

  조건: Booking.status = PENDING 또는 ON_HOLD
        AND isPaid = false
        AND 희망 레슨일까지 남은 일수 체크

  D-2 리마인더 (코치에게만):
    스케줄러: 매일 09:00 실행
    조건: 희망 레슨일 - 2일 AND status IN (PENDING, ON_HOLD)
    → 코치에게 PENDING_CONFIRM_REMINDER 알림톡 발송
    → "아직 확정하지 않은 레슨 신청이 있어요. 내일까지 확정하지 않으면 수강생에게 안내됩니다."

  D-1 최종 리마인더 (코치 + 수강생):
    스케줄러: 매일 09:00 실행
    조건: 희망 레슨일 - 1일 AND status IN (PENDING, ON_HOLD)
    → 코치에게 PENDING_CONFIRM_FINAL 알림톡:
      "레슨 신청이 아직 미확정입니다. 오늘 중 확정 또는 거절해 주세요."
    → 수강생에게 PENDING_CONFIRM_FINAL 알림톡:
      "코치님이 아직 레슨을 확정하지 않았어요. 코치님에게 문의하거나 다른 코치를 찾아보세요."

  희망 레슨일 당일 자동 처리:
    조건: 희망 레슨일 당일 AND status IN (PENDING, ON_HOLD)
    → 자동 처리 없음 (코치 판단에 맡김)
    → 단, 대기 신청 목록에서 "미확정 D-day" 강조 표시
```

### 4.1a 결제 회원 vs 미결제 회원 기능 차이

| 기능 | 미결제 (신청만) | 결제 후 신청 |
|------|:-------------:|:-----------:|
| 레슨 신청 | ✓ | ✓ |
| 코치 수락 필요 | ✓ (PENDING → CONFIRMED) | ✗ (바로 CONFIRMED) |
| 서비스 내 스케줄 관리 | ✗ | **✓** |
| 레슨 이력 조회 | 코치 수락 후 | **즉시** |
| 코치 메모 확인 | 코치 수락 후 | **즉시** |
| D-1 리마인더 알림 | 코치 수락 후 | **즉시** |
| 스케줄 자동 블록 | 코치 수락 시 | **결제 즉시** |
| 환불 | — | 취소 시 환불 처리 |

### 4.1b 환불 정책

#### 정규레슨 환불

```
레슨 시작 기준:
  3일 전 취소 → 전액 환불 (100%)
  1~2일 전 취소 → 90% 환불 (위약금 10%)
  당일 취소 → 환불 불가 (0%)

처리:
  수강생 마이페이지 → 신청 내역 → 취소 버튼
  → 환불 금액 안내 + 확인 다이얼로그
  → POST /api/payments/:id/refund { refundAmount, refundReason }
  → 토스페이먼츠 부분/전액 환불 API 호출
  → Booking.status = CANCELLED
  → Payment.status = REFUNDED, refundedAt = NOW()
  → 코치 스케줄 블록 해제
```

#### 쿠폰레슨 환불

```
미사용 횟수 비례 환불:
  환불 금액 = 결제 금액 - (사용 횟수 × 1회당 금액)
  1회당 금액 = 결제 금액 / 총 횟수

예시:
  10회 쿠폰 ₩500,000 결제, 3회 사용 시:
  1회당 = ₩50,000
  환불 = ₩500,000 - (3 × ₩50,000) = ₩350,000

유효기간 만료 후:
  → 환불 불가
  → 토스트: "유효기간이 만료되어 환불이 불가합니다"
```

#### 코치 책임 취소 (공통)

```
코치 노쇼, 코치 사유 취소 시:
  → 100% 전액 환불 (위약금 없음)
  → 코치에게 경고 알림 발송
  → 반복 시 (3회 이상) 관리자에게 알림 → 조치 검토
```

#### 환불 불가 케이스

```
1. 레슨 완료 (COMPLETED) 후 → 환불 불가 (리뷰로 해결 유도)
2. 쿠폰 유효기간 만료 후 → 환불 불가
3. 정규레슨 당일 수강생 취소 → 환불 불가
```

### 4.2 중복 신청 방지

```
동일 코치에게 동일 요일+시간 조합으로 PENDING 상태 신청이 이미 있으면:
  → 신청 거부 + "이미 해당 시간에 신청한 내역이 있습니다" 토스트
```

## 5. 스케줄 관리 로직

### 5.1 슬롯 단위

- 10분 단위 슬롯: "09:00", "09:10", "09:20", ..., "22:50"
- 1일 최대 슬롯: 102개 (06:00~22:50)

### 5.1a 공통 슬롯 노출 원칙 (Single Source of Truth)

> **모든 일정 선택 화면**에서 노출되는 시간 슬롯은 이 원칙을 따른다.
> 이 섹션의 규칙이 우선 적용되며, 각 플로우 섹션은 이를 참조한다.

```
[기본 원칙]
  코치 측 등록 슬롯 중 "현재 시점에 비어있는 슬롯"만 수강생/코치에게 노출
  = 등록됨 AND 미점유 AND 미잠금 AND 운영가능

[필터링 4단계]
  ① 등록 여부 (Schedule)
     - 코치가 1-7 스케줄 등록 화면에서 추가한 슬롯만 후보
     - dayOfWeek + slotTime (반복) 또는 specificDate + slotTime (단일)

  ② 예약 점유 여부 (Booking + Schedule.isBlocked)
     - isBlocked = true 인 슬롯 제외
     - status IN (PENDING, CONFIRMED, MAKEUP_PENDING, MAKEUP_CONFIRMED) 인 Booking이 점유한 슬롯 제외
     - status IN (CANCELLED, REJECTED, COMPLETED, ABSENT) 의 Booking은 슬롯 점유 해제 처리

  ③ 홀드 잠금 여부 (ScheduleHold)
     - 다른 변경 요청에서 "모두 제안하기"로 임시 홀드된 슬롯 제외
     - 조건: ScheduleHold WHERE expiresAt > NOW() AND coachId = {코치ID}
     - 만료된 홀드는 자동 해제 → 다시 노출

  ④ 운영 정책 필터
     - 코치 휴무일/임시 블록 (ScheduleException) 제외
     - 어드민이 강제 블록한 슬롯 제외
     - 과거 시간 제외 (D-1 이전 등 정책 기한)
     - 영업 시간 외 제외 (06:00~22:50)

[적용 화면 (필수)]
  • 신규 레슨 신청 (3-1)               — 수강생이 코치 가용 시간 선택
  • 수강생 변경 요청 풀팝업 (6-6)        — 수강생이 희망 시간 다중 선택
  • 코치 변경 요청 응답 풀팝업 (7-0)     — 코치가 제안할 가능 시간 선택
  • 코치 보강 날짜 시트 (7-0)           — 코치가 보강 일정 다중 선택
  • 어드민 강제 조치 슬롯 선택 (9-9)    — 어드민이 강제 수락/완료 시
  • 코치 상세 공개 캘린더 (2-3)         — 비로그인 사용자도 가용 슬롯 확인

[예외: 노출은 하되 비활성 처리]
  - 본인이 이미 신청한 슬롯 (중복 방지) → 회색 + "이미 신청됨" 라벨
  - 본인의 다른 레슨이 잡혀 있는 슬롯 → 회색 + "내 일정 있음" 라벨
  - 어드민 강제 블록 슬롯 → 회색 + "운영 차단" 라벨

[가용 슬롯 0개일 때 안내]
  공통 문구: "⚠️ 이 날짜에는 코치님의 가능한 시간이 없습니다.
            다른 날짜를 선택해 주세요"

[API 표준]
  GET /api/coaches/:id/available-slots
    Query: ?date=YYYY-MM-DD (또는 ?from=&to=)
           &purpose=new|reschedule|makeup (요청 목적별 추가 필터 적용)
           &excludeBookingId={id} (변경요청 시 본인 기존 예약 제외)
    Response: { date, slots: [{ time, available, reason? }] }
              available=false인 경우 reason: 'booked'|'held'|'blocked'|'past'

[캐시]
  - 가용 슬롯은 30초 캐시 (Redis)
  - Booking/ScheduleHold INSERT/UPDATE 시 즉시 무효화
  - 동시성 제어: 슬롯 선택 시 서버에서 최종 검증 (낙관적 잠금)
```

### 5.1b 코치 스케줄 홈 (7-0) UI 규칙

```
타이틀 영역:
  "나의 스케줄" (20px, 상단) + "안녕하세요, {코치명} 코치님" (12px, 하단) 순서 배치

주간 미니 캘린더:
  날짜별 점 표시:
    - 초록(●): 해당 날짜에 레슨이 있는 날 (CONFIRMED/COMPLETED)
    - 보라(●): 해당 날짜에 변경 요청이 있는 날 (rescheduleStatus=PENDING/PROPOSED)
    - 주황(●): 해당 날짜에 대기 중 신청이 있는 날 (status=PENDING/ON_HOLD)
    - 복수 상태 시 점 다중 표시 (최대 3개)
    - 오늘(활성 셀)은 점 색상 흰색으로 변경

  날짜 클릭 동작:
    - 클릭한 날짜 셀 초록 활성화 → 나머지 셀 비활성 (gray-50)
    - 상단 날짜 라벨 동적 변경 ("📅 {날짜}")
    - **모든 탭이 선택 날짜에 연동** (위계상 캘린더 > 탭):
      오늘 레슨: 해당일 확정/완료 건
      변경 요청: 해당일 기존 일정 기준 건 (상단 "{날짜} 기존 일정 기준" + "전체보기 ›")
      대기 신청: 해당일 희망 일정 기준 건 (상단 "{날짜} 희망 일정 기준" + "전체보기 ›")
    - 레슨 없는 날: 빈 상태 메시지 ("레슨이 없는 날이에요")
    - 오늘 다시 클릭 시 원래 콘텐츠 복원
    - API:
      오늘 레슨: GET /api/bookings?role=coach&date={선택날짜}&status=CONFIRMED,COMPLETED
      변경 요청: GET /api/bookings?role=coach&date={선택날짜}&rescheduleStatus=PENDING,PROPOSED
      대기 신청: GET /api/bookings?role=coach&desiredDate={선택날짜}&status=PENDING,ON_HOLD
    - 전체보기: 날짜 필터 없이 전체 목록 (7-2-booking-inbox.html)

  범례 툴팁:
    - 날짜 라벨 우측 ? 버튼 클릭 시 네이비 툴팁 표시
    - 초록=레슨 있는 날 / 보라=변경 요청 / 주황=대기 중 신청
    - 바깥 클릭 시 닫힘

상태 안내 바:
  - 그레이 박스 (gray-100) 한 줄: 🎾진행중 📌예정 🔄변경 🏁완료 + 우측 ?
  - 클릭 시 상태 설명 팝업 (각 상태별 아이콘 + 제목 + 친절한 설명)

대기 중 신청 카드:
  - 카드에 신청 날짜 표시 ("{날짜} 신청 · {요일} {시간} 희망")
  - 클릭 시 레슨 상세 풀팝업 오픈 (대기 상태)

오늘 레슨 목록 표시 상태:
  - 🎾 진행중 (코랄/빨강): status=CONFIRMED이고 현재 시간이 레슨 시작~종료 시간 범위 내인 건
    판정: now >= startTime AND now < endTime
    endTime = startTime + lessonDurationMinutes (기본 60분)
  - 📌 레슨 예정 (보라색): status=CONFIRMED이고 오늘 날짜이며 시작 시간이 현재 시간 이후인 건
  - 🔄 변경 요청 (주황색): 오늘 레슨 중 rescheduleStatus가 PENDING 또는 PROPOSED인 건
    변경 요청이 있는 레슨도 오늘 레슨 목록에 함께 표시
  - 🏁 레슨완료 (파란색): status=COMPLETED이고 오늘 날짜인 건
  - ❌ 결강 (회색): status=ABSENT이고 오늘 날짜인 건. 취소선+반투명 처리
  - 🔄 보강 일정 선택중 (초록 점선): status=MAKEUP_PENDING이고 오늘 날짜인 건. 보강 레슨 미확정 상태
  - ✅ 보강확정 (초록 실선): status=MAKEUP_CONFIRMED이고 오늘 날짜인 건. 수강생이 보강 일정 수락 완료
  - ※ "확정" 배지는 오늘 레슨 목록에 노출하지 않음 (레슨 예정/진행중으로 대체)

레슨 시간 표시: 시작 시간만 표시 (종료 시간 미노출)

레슨 상세 풀팝업:
  - 레슨 목표 항목은 표시하지 않음
  - 레슨 정보 표시 항목: 레슨 유형 / 레슨 형태 / 현재 회차 / 결제 상태 / 다음 결제일

  현재 회차 정책:
    표시 형식: "N/M회"
    카운트 기준: 레슨 진행 여부 (결제 여부와 무관)
    - 정규레슨: 이번 달 진행 회차(완료+예정) / 월 총 횟수
      월 총 횟수 = weeklyCount × 해당 월 잔여 주수
      - 월 초부터 시작: weeklyCount × 4주 (예: 주2회 → 8회)
      - 월 중간 시작 (첫 달): weeklyCount × 잔여 주수 (예: 4/15 시작 주2회 → 2주 × 2 = 4회)
      - 다음 달부터: weeklyCount × 4주 (정상 복귀)
      N = 이번 달 status=COMPLETED 건수 + 오늘 이후 CONFIRMED 건수
    - 쿠폰레슨: 전체 사용 회차 / 총 구매 회차
      N = status=COMPLETED 건수, M = BookingCouponDetail.totalSessions
    표시 위치: 오늘 레슨 리스트 카드 + 레슨 상세 풀팝업

  결제 상태 정책:
    - 서비스 결제: isPaid=true, paymentId 존재 (토스페이먼츠 결제 완료). 초록색 "결제완료" 표시
    - 외부 결제: isPaid=true, paymentId=null (코치가 상태처리하기에서 확인). 파란색 "외부결제" 표시
      트리거: 상태처리하기 바텀시트 → "결제 확인" (미결제일 때만 노출)
      → 결제 수단 선택 (현금/계좌이체/기타) → 금액 확인 → 확인
      → isPaid=true, paymentMethod='EXTERNAL', externalPaymentType='CASH'|'TRANSFER'|'OTHER'
      → 결제 상태 "외부결제"로 전환
    - 미결제: isPaid=false. 빨간색 "미결제" 표시
    ※ 결제 상태와 무관하게 레슨 진행·회차 카운트 가능 (코치 재량)
    ※ 리스트 카드에서 미결제 건은 "미결제" 배지로 코치에게 미수금 알림

  최근 결제일: 해당 수강생의 가장 마지막 결제 완료 날짜 (Payment.paidAt 기준, 서비스 결제+외부 결제 모두 포함)
    - 결제 이력이 없으면 미표시
    - 서비스 결제: Payment.paidAt
    - 외부 결제: 코치가 외부 결제 확인 처리한 날짜
  다음 결제일: 정규레슨 월 결제 기준 다음 결제 예정일 표시 (쿠폰레슨은 미표시)
  - [변경 요청] 상태: 레슨 정보(유형/형태/결제) 하단에 "변경 요청" 타이틀로 영역 구분 후
    일정 통합 박스: 주황 배경(#FFF7ED) 1개 박스에 기존일정(취소선) → 희망날짜 표시 + "수강생 희망 변경" 라벨
    수강생 희망 날짜가 코치 가용 시간에 포함되면 통합 박스 우측에 [확정하기] 버튼 표시 (코랄색 강조 + box-shadow)
    확정하기 클릭 → 확인 다이얼로그 → 해당 희망 날짜로 즉시 변경 확정 (제안 단계 생략)
    → rescheduleStatus=COMPLETED + 기존 스케줄 해제 + 새 스케줄 블록 + 수강생 알림
    추천 슬롯 세로 리스트 (flex-direction:column): 체크박스 + 날짜 (다중 선택 가능)
    희망 날짜 가용 시 추천 목록에서 동일 슬롯 제거 (중복 방지), 타이틀 "변경 가능한 다른 날짜"
    희망 날짜 미가용 시 전체 추천 목록 표시, 타이틀 "변경 가능한 날짜"
    하단 버튼 좌/우 배치: [선택 제안하기] (선택한 슬롯만 제안, 선택 시 활성화) | [모두 제안하기] (전체 슬롯 제안)
    선택 제안하기: 아웃라인 스타일 (초록 테두리+텍스트), 선택 없으면 비활성(회색)
    버튼 하단에 제안하기 안내 영역: 제안 시 12시간 임시 홀드(해당 시간대 다른 변경 요청에서 추천되지 않음) + 수강생에게 알림톡 발송 + 응답 없으면 자동 해제 설명
    추천 슬롯이 없으면 "불가 알림 보내기" 버튼 표시
  - 상태별 하단 액션:
    [진행중]   상태처리하기
    [레슨 예정] 상태처리하기
    [대기 중]   더보기(보류/거절/고객과 직접 일정잡기) + 스케줄 확정하기 (상태처리하기 버튼 비노출)
    [변경 요청] 상태처리하기(좌측, 아웃라인) + 제안하기(우측, 선택 0개면 비활성)
    [레슨완료]  상태처리하기 + 닫기
    [결강]     상태처리하기(결강처리 비노출) + 닫기
    [보강 일정 선택중]  상태처리하기 + 닫기
    [보강확정]  상태처리하기 + 닫기

  변경 요청 — "변경 가능한 날짜" 영역 노출 규칙:
    - 라벨: 수강생 희망 날짜가 가용하면 "변경 가능한 다른 날짜", 아니면 "변경 가능한 날짜"
    - AI 이모지(🤖) 사용 금지 — 자동 추천 결과지만 시각적으로 AI 강조 안함
    - 일/주 토글로 보기 모드 전환:
      [일 모드] (기본): 1개 날짜 + 해당 날짜의 시간 슬롯 세로 리스트
        ◆ 디폴트 선택: 오늘 날짜 (실시간)
        ◆ 오늘 가용 슬롯이 0개면 → 오늘 이후 가장 가까운 가용 날짜로 자동 선택 (최대 14일 탐색)
        ◆ 좌우 화살표로 ±1일 이동 (오늘 이전 비활성)
      [주 모드]: 7일 윈도우 + 날짜별 가용 시간 칩 표시
        ◆ 디폴트 윈도우: 오늘부터 7일
        ◆ 좌우 화살표로 ±7일 이동 (이전 주 비활성)
    - 슬롯 다중 선택 (체크박스, 사각형 — 디자인 가이드 6.5a 준수)
    - 선택된 슬롯은 "M/D HH:MM" 키로 메모리 관리
    - 선택 0개: "제안하기" 버튼 비활성(회색)
    - 선택 1개 이상: "제안하기 (N개)" 버튼 활성화(초록)
    - 수강생 희망 날짜 박스의 "확정하기" 버튼: 희망 날짜가 가용 슬롯이면 코랄색 강조 노출 (별도 플로우)

  상태처리하기 바텀시트 항목:
    🏁 레슨 완료 — 정상 완료 처리
    ➕ 보강 처리 — 추가 보강 레슨 등록 (변경요청 상태에서는 비노출)
    ✕ 결강 처리 — 결강 처리 + 수강생 알림 (결강 상태에서는 비노출)
    💰 결제 확인 — 외부 결제 확인 (미결제일 때만 노출)
      → 결제 수단 선택 바텀시트 (현금/계좌이체/기타)
      → 금액 확인 → 저장 → 외부결제 전환

  상태별 상태처리하기 바텀시트 버튼 노출 규칙:

  | 상태 | 레슨 완료 | 보강 처리 | 결강 처리 | 진행 예정 되돌리기 | 결제 확인 |
  |------|:--:|:--:|:--:|:--:|:--:|
  | 진행중 | ✓ | ✓ | ✓ | — | (미결제 시) |
  | 레슨 예정 | ✓ | ✓ | ✓ | — | (미결제 시) |
  | 대기 중 | **✗** | **✗** | **✗** | — | — | (상태처리하기 비노출 — 결제 확인 후 확정 플로우 사용) |
  | 변경 요청 | ✓ | **✗** | ✓ | — | (미결제 시) |
  | 레슨 완료 | ✓ | ✓ | ✓ | — | (미결제 시) |
  | 결강 | ✓ | ✓ | **✗** | — | (미결제 시) |
  | **보강 일정 선택중** | **✗** | **✗** | ✓ | **✓** | (미결제 시) |
  | **보강확정** | ✓ | **✗** | ✓ | — | (미결제 시) |
  | 변경완료 | ✓ | ✓ | ✓ | — | (미결제 시) |

  비노출 사유:
    - **대기 중 + 상태처리하기 전체**: 아직 확정 전이므로 상태 처리 불필요. 더보기(보류/거절) + 스케줄 확정하기로 대체
    - 변경요청 + 보강처리: 기존 일정 변경이 우선 — 변경 요청 응답 후에야 보강 의미 생김
    - 결강 + 결강처리: 이미 결강 상태이므로 중복 처리 불가
    - **보강 일정 선택중 + 레슨완료**: 보강 일정이 아직 확정되지 않아 완료 처리 불가
    - **보강 일정 선택중 + 보강처리**: 이미 보강 진행 중이므로 추가 보강 등록 불가
    - **보강 일정 선택중 + 결강처리는 가능**: 보강 일정 자체가 다시 결강될 수도 있으므로 노출 유지
    - **보강확정 + 보강처리**: 이미 보강 일정이 확정되어 있어 추가 보강 등록 불가
      → 추가 보강 필요 시: 수강생이 6-6에서 "재보강 요청" 시작 → 상태가 makeup_requested로 전환

  진행 예정 되돌리기 (보강 일정 선택중 상태 전용):
    - 코치가 보강 일정을 등록했다가 재검토 필요 시 되돌리기
    - 동작: status MAKEUP_PENDING → CONFIRMED (원래 일정 복원)
    - 영향:
      ① ScheduleHold 전체 삭제 (제안된 보강 슬롯 잠금 해제)
      ② 원래 결강된 일정의 status를 CONFIRMED로 복원
      ③ 수강생에게 알림톡 발송 ("보강 제안이 취소되었습니다. 코치님이 재검토 중입니다.")
      ④ RescheduleActionLog/MakeupActionLog에 기록 (actionType: REVERT_MAKEUP)
    - 사용 예: 코치가 보강 일정 잘못 등록 / 다른 사정으로 보강 보류
    - 보강 절차 재시작: 다시 결강 처리 후 보강 등록 가능

  전체 스케줄 보기 — 과거/미래 슬롯 처리:
    - 과거 레슨 슬롯: 자동으로 "완료" 상태 변환 + opacity 0.5
    - 과거 빈 슬롯: gray-100 비활성 + 선택 불가 + 토스트 안내
    - 미래 빈 슬롯: 다중 선택 가능 (초록 배경)
    - 레슨 슬롯 클릭: 올리브그린 토스트 (수강생 정보 + "레슨 상세 보기 ›" 버튼)
```

### 5.1c 레슨 완료 처리 정책

```
레슨 완료 처리는 두 가지 방식으로 이루어짐:

1. 수동 완료: 코치가 레슨 상세에서 "레슨 완료 처리" 버튼 클릭
   → 확인 다이얼로그 → PUT /api/bookings/:id/complete
   → status = COMPLETED + 완료 시각 기록

2. 자동 완료: 다음 레슨 시작 시각 도래 시 이전 레슨 자동 완료
   판정: 동일 코치의 다음 CONFIRMED 건의 startTime에 도달했을 때
         이전 CONFIRMED 건이 아직 COMPLETED가 아니면 자동 전환
   → 스케줄러(Cron) 또는 다음 레슨 조회 시 트리거
   → status = COMPLETED + completedAt = 해당 레슨의 endTime
   → 자동 완료 시 코치에게 "레슨이 자동 완료 처리되었습니다" 알림

자동 완료 실행 시점:
  - 방법 A (Cron): 매 10분마다 체크
    SELECT * FROM Booking
    WHERE status = 'CONFIRMED'
      AND coachId IN (
        SELECT DISTINCT coachId FROM Booking
        WHERE status = 'CONFIRMED' AND startTime <= NOW()
      )
    ORDER BY coachId, startTime ASC
    → 같은 코치의 다음 건 startTime <= NOW()이면 이전 건 자동 COMPLETED

  - 방법 B (이벤트): 코치 스케줄 홈 조회 시 (GET /api/bookings?role=coach&date=today)
    서버에서 오늘 레슨 목록 반환 전, 자동 완료 대상 체크 후 상태 업데이트

※ 자동 완료된 레슨도 회차에 정상 카운트됨
※ 당일 마지막 레슨은 자동 완료 대상이 아님 (코치 수동 처리 필요)
```

### 5.1c-2 스케줄 확정 시 결제 확인 정책

```
대기 중(PENDING) 상태에서 코치가 "스케줄 확정하기" 클릭 시
→ 결제 확인 3지선다 모달 노출 → 선택 후 확정

[옵션 3종]
  ① 전액 받음 (paid)
     - paymentStatus = paid
     - "결제완료" 배지 표시
     - 수강생 알림톡: 일반 확정 알림 (결제 확인됨)

  ② 예약금만 받음 (deposit)
     - paymentStatus = deposit
     - "예약금" 배지 표시
     - 수강생 알림톡: 확정 + 잔액 안내 ("잔액 X원은 첫 레슨 시 결제해 주세요")

  ③ 아직 미결제 (unpaid)
     - paymentStatus = unpaid
     - "미결제" 배지 표시 (코랄)
     - 수강생 알림톡: 확정 + 결제 안내 ("결제는 첫 레슨 시 정산해 주세요")
     - 코치 측 7-0 풀팝업에 "결제 확인" 버튼 자동 노출 (추후 결제 받으면 갱신)

공통:
  - status: PENDING → CONFIRMED 전환
  - 수강생 희망 일정 1개로 자동 확정 (라디오 선택 일정)
  - 다른 일정 슬롯 점유 해제
  - 알림톡: BOOKING_CONFIRMED (paymentStatus 별 분기 메시지)

UI 구성:
  - 모달 위치: 풀팝업 내부 z-index 160
  - 헤더: "스케줄을 확정하기 전에"
  - 본문: "{수강생명}님으로부터 결제를 받으셨나요?"
  - 옵션 3개 카드 (✅ 초록 / 💵 주황 / ⏳ 빨강)
  - 하단 "취소" 버튼

API:
  PUT /api/bookings/:id/confirm
  { selectedScheduleId, paymentStatus: 'paid' | 'deposit' | 'unpaid' }
```

### 5.1d 결강 / 보강 정책

```
결강(ABSENT) = 노쇼 (예정된 레슨이 진행되지 않음):
  결강 사유에 따라 회차 차감 / 보강 가능 / 환불 가능 여부가 달라짐
  ※ 모든 정책 기본값은 어드민 9-9 "결강 정책 (노쇼)" 설정에서 조정 가능

결강 유형 (5종) — 어드민 정책 매트릭스:

  | 결강 유형 (사유) | 회차 차감 | 보강 가능 | 환불 가능 | 비고 |
  |-----------------|:---------:|:---------:|:---------:|------|
  | 수강생 노쇼 (무단) | ✓ | ✗ | ✗ | 수강생 책임, 종결 |
  | 수강생 사전 통보 (D-1 이후) | ✓ | ✗ | ✗ | 임박 통보, 회차 차감 |
  | 수강생 사전 통보 (D-2 이전) | ✗ | ✓ | △ (정책별) | 충분한 사전 통보 — 보강 가능 |
  | 코치 사정 | ✗ | ✓ | △ | 회차 보존 + 보강 권장 |
  | 천재지변/공동 휴무 | ✗ | ✓ | △ | 어드민 검토 |

  ※ ✓ = 기본 활성, ✗ = 기본 비활성, △ = 어드민 토글
  ※ "사전 통보 기준일" 기본 D-2 (어드민에서 조정 가능, 1~7일)

결강 처리 트리거:
  - 코치가 7-0 레슨 상세 → 상태처리하기 → "결강 처리" 클릭
  - 결강 사유 입력 화면:
    ① 결강 유형 (라디오, 필수 — 위 5종 중 1개)
    ② 결강 사유 (텍스트, 필수)
  - status: CONFIRMED → ABSENT
  - 어드민 정책 매트릭스 따라 자동 처리:
    · 회차 차감/보존 결정
    · 보강 등록 화면 노출 여부
    · 환불 정책 적용 여부
  - 수강생에게 알림톡 발송 (BOOKING_ABSENT) — 결강 유형별 메시지 분기

보강 등록 — 양방향 모두 가능 (단, 어드민 정책에서 "보강 가능"인 결강 유형만):

[A. 코치 능동적 보강 등록 — 권장 기본 흐름]
  - 코치가 결강 처리 시 바로 "보강 일정 등록" 화면으로 유도
  - 코치가 가능한 일정 1개 이상 선택 → 수강생에게 제안
  - status: ABSENT → MAKEUP_PENDING
  - 수강생이 1개 일정 선택 시: MAKEUP_PENDING → MAKEUP_CONFIRMED
  - 수강생 알림톡: MAKEUP_PROPOSED → MAKEUP_CONFIRMED

[B. 수강생 능동적 보강 요청 — 안전망]
  - 결강 후 코치가 24시간 내 보강 미등록 시 자동 활성
  - 수강생 6-6 결강 화면에 "보강 요청하기" 버튼 노출
  - status: ABSENT → MAKEUP_REQUESTED
  - 코치에게 알림톡 발송: 보강 일정 제안 요청
  - 코치가 응답 시: MAKEUP_REQUESTED → MAKEUP_PENDING (제안) 또는 MAKEUP_CONFIRMED (직접 확정)

[제한]
  - 보강 요청은 ABSENT 상태인 레슨에만 가능 (다른 상태 불가)
  - 결강 유형이 "보강 가능"으로 어드민에서 설정된 케이스에만 보강 등록 가능
    · 기본: 수강생 사전통보(D-2 이전) / 코치 사정 / 천재지변
    · 기본 보강 불가: 수강생 노쇼 / 수강생 사전통보(D-1 이후)
  - 1회 결강당 보강 1회만 가능 (중복 보강 불가)
  - 보강 일정은 결강일 기준 ±30일 내로 제한 (어드민 정책 설정 가능)
  - 미결제 회원: 보강 불가 (회차 보호만 적용)

[코치 측 7-0 화면 표시]
  - 결강 카드: 회색 + "❌ 결강" 배지
  - 보강 일정 선택중: 초록 점선 + "🔄 보강 일정 선택중" (코치가 제안한 일정, 수강생 응답 대기)
  - 보강확정: 초록 실선 + "✅ 보강확정" (수강생이 일정 수락)
  - 보강 요청: 주황 테두리 + "🙋 보강 요청" (수강생이 요청, 코치 응답 대기)

[수강생 측 6-5/6-6 화면 표시]
  - 결강: "❌ 결강" + "보강 가능" 안내
  - 보강 요청: "🙋 보강 요청" (수강생이 요청한 상태)
  - 보강 일정 선택중: "🔄 보강 일정 선택중" (코치가 제안, 수강생이 선택해야 함)
  - 보강 확정: "✅ 보강 확정" (확정된 보강 일정)
```

### 5.2a 스케줄 변경 요청

```
수강생이 확정된 레슨 일정 변경 요청:

1. 수강생 요청 — 수강생 레슨 상세(6-6) 화면에서 "변경 요청" 버튼 클릭:
   ① 변경 요청 풀팝업 오픈
   ② 입력 항목:
      - 현재 일정 (자동 표시, 취소선)
      - 희망 날짜 (D+1부터 7일 그리드, 주 단위 이동, 코치 가용 요일만 활성)
      - 희망 시간 (선택한 날짜의 가용 슬롯 칩)
      - 변경 사유 (라디오: 개인 사정 / 일정 충돌 / 컨디션 / 기타) — 필수
      - 추가 메모 (선택)
   ③ 모든 필수 입력 완료 시 "변경 요청 보내기" 활성화
   ④ 확인 모달 → 요약(현재→희망, 사유) + 알림톡 발송 안내 → "요청 보내기"
   ⑤ POST /api/bookings/:id/reschedule
      { requestedDate, requestedTime, reason, reasonDetail }
   → Booking.rescheduleStatus = PENDING
   → 코치에게 변경 요청 알림 (RESCHEDULE_REQUEST 알림톡)
   → 수강생 레슨 상세는 reschedule_pending 상태로 전환 (희망 일정 표시 + "코치 응답 대기" 안내)

   reschedule_pending 상태 표시 (수강생 6-6):
     - 희망 일정 다중 노출 시 **날짜별 그룹핑** (같은 날짜는 시간 칩으로 묶어서 표시)
     - 일정 4개 이상 시 max-height: 180px + 세로 스크롤 + "↓ 스크롤" 안내
     - 헤더에 "희망 변경 일정 N개 · M일" 카운트 표시

2. 코치 응답 후 — reschedule_completed 상태 (수강생 6-6):
   - 배지: ✅ 변경 완료 (파란색)
   - 메시지: "레슨 일정 변경이 완료되었습니다 ✓"
   - 변경 전/후 비교 박스: 기존 일정(취소선) → 확정 일정(강조) + "확정" 배지
   - 변경 사유 표시 (수강생이 입력한 사유)
   - 코치 메시지 (선택적, params.coachMsg가 있을 때만 노출)
   - 안내 문구: "변경된 일정은 회차에 영향을 주지 않으며, 결제일도 변경되지 않습니다."
   - URL 파라미터: status=reschedule_completed, original, confirmedDate, reason, coachMsg
   - 하단 액션: 코치에게 문의 / 목록으로

2. 시스템 자동 추천:
   코치 Schedule 테이블 조회:
     WHERE coachId = {코치ID}
       AND dayOfWeek = {희망 요일}   (또는 ±1일 범위)
       AND isBlocked = false
     ORDER BY slotTime ASC
   → 변경 가능 날짜+시간 목록 반환 (최대 5개)
   
   추천 범위:
     1순위: 수강생 희망 날짜의 동일 시간
     2순위: 수강생 희망 날짜의 다른 시간 (코치 가용)
     3순위: 희망 날짜 ±1일의 가용 시간

3. 변경 가능 여부 판정:
   자동 추천 결과에 따라 두 케이스로 분기:

   [변경 가능] 추천 슬롯 ≥ 1개:
     → 코치 화면에 초록색 "변경 가능" 카드 표시
     → 추천 날짜+시간 목록 (최대 5개) + "모두 제안하기" 버튼

   [변경 불가] 추천 슬롯 = 0개:
     → 코치 화면에 빨간색 "변경 불가" 카드 표시
     → 사유 메시지: "수강생이 희망한 날짜 근처에 변경 가능한 빈 시간이 없습니다."
     → "불가 알림 보내기" 버튼

4. 슬롯 임시 홀드 (하이브리드 정책):
   동시 다수 변경 요청에 의한 슬롯 충돌 방지:

   [홀드 생성]
     코치가 "모두 제안하기" 클릭 시:
       POST /api/bookings/:id/reschedule/propose { slotIds: [...] }
       → ScheduleHold 레코드 생성 (bookingId, slotId[], heldAt, expiresAt)
       → holdHours = CoachProfile.holdDurationHours ?? ScheduleSetting.holdDurationHours (기본 12)
       → expiresAt = heldAt + holdHours시간
       (코치별 개별 설정 우선, 없으면 전역 설정 적용)
       → 수강생에게 제안 알림 (카카오 알림톡)

   [홀드 중 다른 요청 처리]
     새로운 변경 요청의 자동 추천 시:
       WHERE isBlocked = false
         AND slotId NOT IN (SELECT slotId FROM ScheduleHold WHERE expiresAt > NOW())
       → 홀드된 슬롯은 추천 목록에서 제외
       → 남은 가용 슬롯이 0개이면 "변경 불가" 처리
         사유: "다른 변경 요청 처리 중으로 가용 시간이 없습니다."

   [수강생 응답 처리]
     수락: PUT /api/bookings/:id/reschedule/accept { selectedSlotId }
       → 기존 날짜 Schedule isBlocked=false (해제)
       → 선택된 슬롯 Schedule isBlocked=true (블록)
       → Booking 날짜/시간 업데이트
       → rescheduleStatus = ACCEPTED
       → 해당 bookingId의 ScheduleHold 전체 삭제 (나머지 홀드 해제)
       → 홀드 해제된 슬롯을 사용하는 다른 PENDING 변경 요청 자동 재추천 트리거
       → 수강생 알림: "스케줄이 변경되었습니다. {새 날짜} {새 시간}"

     거절 (수강생이 모든 제안 거절):
       PUT /api/bookings/:id/reschedule/reject-proposal
       → ScheduleHold 전체 삭제
       → rescheduleStatus = NEGOTIATING (재협의 필요)
       → 코치 알림: "수강생이 제안을 거절했습니다. 직접 조율이 필요합니다."
       → 채팅으로 유도

   [홀드 자동 만료]
     스케줄러 (CRON / 12시간 주기):
       DELETE FROM ScheduleHold WHERE expiresAt <= NOW()
       → 만료 시 코치 알림: "{수강생명}님이 12시간 내 응답하지 않았습니다."
       → rescheduleStatus 유지 (PENDING) → 코치가 재제안 또는 거절 선택

   [불가 알림 보내기]
     코치가 "불가 알림 보내기" 클릭 시:
       PUT /api/bookings/:id/reschedule/reject { reason: "UNAVAILABLE" }
       → rescheduleStatus = REJECTED
       → 기존 일정 유지
       → 수강생 알림: "변경 가능한 시간이 없어 기존 일정으로 진행됩니다."

5. 변경 완료 처리:
   수강생이 제안 수락 완료 후:
     → rescheduleStatus = COMPLETED
     → 코치 스케줄 홈에 "변경 완료" 카드로 표시 (당일까지)
     → 변경 전/후 일정 모두 표시 (확인용)
     → 다음날 00:00 이후 목록에서 자동 제거

변경 요청 정렬 규칙:
  먼저 요청한 수강생이 상단 노출 (요청일시 ASC — 선입선출)
  ORDER BY Booking.rescheduleRequestedAt ASC

  상태별 그룹 없이 순수 요청 시각순 정렬:
    → 먼저 요청한 수강생이 먼저 처리될 수 있도록 유도
    → 코치가 위에서 아래로 순서대로 처리하는 자연스러운 UX

변경 요청 제한:
  - 레슨 당일 변경 요청 불가 (D-1까지만)
  - 동일 예약에 대해 PENDING 상태 변경 요청 1건만 가능
  - 결제 건(isPaid=true)만 변경 요청 가능 (미결제는 취소 후 재신청)
  - 홀드 유효기간: 12시간 (만료 후 자동 해제)
  - 수강생 1인당 동시 홀드 가능 건수: 1건
  - 희망 일정 선택 가능 범위: **당월(현재 월) 내**로 제한 — 다음 달 이후 선택 불가
  - 수강생 1인당 월 변경 요청 횟수: **최대 3회**(설정값 maxRescheduleRequestsPerMonth)

당월 제한 정책 (희망 일정 선택 범위):
  [수강생 — 변경 요청 풀팝업]
    - 일/주 토글 모두 당월 내 날짜만 활성
    - 일 모드: 오늘(리얼타임) ~ 당월 마지막 날(매월 28~31일)까지 좌우 화살표로 이동
              당월 마지막 날 도달 시 다음(›) 버튼 비활성
    - 주 모드: 7일 그리드, 당월 범위 밖 날짜는 비활성 셀 처리(opacity 0.4)
              다음 주 시작점이 다음 달이면 다음(›) 버튼 비활성
    - 다음 달로 변경 필요 시 안내: "다음 달 일정은 변경 요청이 불가합니다.
      해당 일정 발생 후 새로 신청해 주세요."
    - **디폴트 활성 날짜**: 오늘(리얼타임). 오늘 가용 슬롯 0개면 당월 내 가장 가까운 가용 날짜 자동 선택
    - **희망 일정 영역 통합 박싱** (날짜 + 시간 한 박스):
      ① 외곽: 초록 배경(#F0FDFA) + 초록 보더(#D1FAE5) — 단일 그룹
      ② 상단: 일/주 토글 + 날짜 네비게이션
      ③ 내부 구분선(#D1FAE5)
      ④ 하단: "가능한 시간 (중복 선택 · 총 N개)" 라벨 + 체크박스 시간 리스트
    - **시간 다중 선택 (체크박스)**:
      - 단일 라디오가 아니라 **체크박스로 다중 선택** 가능
      - 수강생이 여러 후보를 보내면 코치가 그 중 가능한 시간으로 확정 (또는 다른 시간 제안)
      - 선택된 슬롯 메모리 구조: Map<dateKey, Set<time>>
      - 다른 날짜로 이동해도 이전 선택 유지 — "총 N개" 카운트 표시
    - **하단 안내 문구**: "💡 코치님이 검토 후 응답해 드립니다.
      요청하신 일정 중 가능한 시간으로 확정되며, **모두 어려운 경우 코치님이 다른 가능한 일정을 제안해 드릴 수 있어요.**
      코치 응답은 카카오 알림톡으로 안내드립니다."
    - **시간 노출 로직**: 코치가 최초 등록한 시간대(CoachSlot) 중 다음 조건 모두 충족하는 것만:
      ① 해당 날짜에 다른 레슨 예약(Booking)으로 점유되지 않음
      ② 다른 변경 요청 홀드(ScheduleHold)에 잠기지 않음
      ③ 코치 휴무일/블록 시간이 아님
    - **가용 시간 0개 시 안내**: "⚠️ 이 날짜에는 코치님의 가능한 시간이 없습니다. 다른 날짜를 선택해 주세요"
    - **확인 모달**: 선택된 모든 일정을 리스트로 미리보기 (스크롤 처리 max-height 240px)
    - **제출 시 전달**: status=reschedule_pending, desired(첫 일정), allDesired(전체 파이프 구분)

  [수강생 변경 요청 횟수 표시]
    - 수강생 레슨 상세(6-6) 공통 정보 박스에 항상 표시: "이번 달 변경 요청  N / M회"
      (M = ScheduleSetting.maxRescheduleRequestsPerMonth, 어드민 9-9 정책 설정에서 조정)
    - reschedule_pending 상태 안내 박스 하단에도 카운트 표시
    - 변경 요청 풀팝업 상단에도 카운트 박스 (노란 배경)

  [코치 변경 요청 횟수 표시]
    - 코치 레슨 상세(7-0 풀팝업) 공통 정보 박스에 항상 표시: "이번 달 변경 요청  N / M회"

월 횟수 제한 정책:
  [공통]
    - 기준: 매월 1일 00:00에 카운트 리셋

    - reschedule_pending 상태 안내 박스 하단에도 동일 카운트 표시

    - 7-0 풀팝업 변경 요청 헤더 우측: "수강생 이번 달 N / 3회" 칩

    - 안내 박스 디자인: 배경 #FEF2F2 + 보더 #FECACA + ⚠️ 아이콘 + 빨간 텍스트(#991B1B)
    - 시스템에서 자동 차단 (서버/클라이언트 양쪽 검증)

  [코치 측]
    - 별도 안내 불필요 (수강생 측에서 시스템 차단되므로 변경 요청 자체가 들어오지 않음)

  [어드민 측]
```

## 6. 리뷰/평점 계산

### 6.1 평균 평점 계산

```

3. 작성자 = 해당 예약의 studentId

5. 텍스트: 최소 10자
```

## 8. 약관 버전 관리

### 8.1 약관 개정 플로우

```
어드민이 약관 개정 시:
  1. Terms 테이블에 동일 code로 새 레코드 생성 (version 증가)
  2. 기존 동일 code의 isActive = false
  3. 신규 레코드 isActive = true
```

### 8.2 재동의 요청 로직

```
사용자 로그인 시:
  1. GET /api/terms (isActive=true인 필수 약관 목록)
  2. GET /api/users/me/terms (사용자가 동의한 termsId 목록)
  3. 비교:
     필수 약관의 현재 활성 termsId ≠ 사용자 동의 termsId
     → 불일치 약관에 대해 재동의 바텀시트 표시
     → 동의 완료 시 UserTermsAgreement 갱신 (새 termsId로)
```

### 8.3 미동의 시 서비스 제한

```
필수 약관 미동의 상태:
  → 서비스 이용 차단 (재동의 바텀시트 닫기 불가)
  → 동의하지 않으면 로그아웃 처리
```

## 9. 인증/권한 정책

### 9.1 API 접근 구분

| 구분 | 인증 | 대상 |
|------|:----:|------|

| **인증 필요** | JWT | 프로필 수정, 예약, 결제, 채팅, 찜 |

### 9.2 비로그인 허용 범위

| 기능 | 비로그인 |
|------|:--------:|
| 홈 화면 진입 | ✓ |
| 코치 검색/목록 | ✓ |
| 코치 상세 조회 | ✓ |
| 리뷰 조회 | ✓ |
| 레슨 신청 | ✗ → 로그인 유도 |
| 문의하기 (채팅) | ✗ → 로그인 유도 |
| 찜하기 | ✗ → 로그인 유도 |
| 메시지 탭 | ✗ → 로그인 유도 |

### 9.3 로그인 시 미가입 처리

```
소셜 로그인 API 응답:
  isNewUser = false → JWT 발급 → 홈
  isNewUser = true → 바텀시트 (약관은 프로필 등록에서) → 역할 선택
  isActive = false → "이용이 제한된 계정입니다" 토스트
```

### 9.4 차단 계정 정책

```
어드민이 User.isActive = false 설정 시:
  → 해당 사용자 JWT 무효화 (블랙리스트 또는 다음 요청 시 체크)
  → 로그인 시도 시 "이용이 제한된 계정입니다. 고객센터에 문의하세요" 토스트
  → 코치 차단 시: 코치 검색/목록에서 제외, 기존 예약은 유지
```

```
     - 채팅 목록에서 해당 채팅방: opacity 40% + 🚫 아이콘 + "차단한 코치의 채팅방입니다" 문구
     - 비활성 채팅방은 목록 하단으로 이동
     - 메시지 전송 불가

  2. 레슨 신청 차단:
     - 해당 코치에게 레슨 신청 시 → "차단한 코치에게는 신청할 수 없습니다" 토스트

  3. 코치 프로필 접근:
     - 검색/목록에서는 표시 (차단 여부 무관)
     - 코치 상세에서 문의하기/레슨신청 버튼 비활성화

차단 해제:
  - 채팅 목록 → 차단된 채팅방 클릭 → "차단 해제" 바텀시트 → 해제 시 정상 복원
```

```
스케줄 히스토리:
```

### 13a.1 빈 슬롯에서 레슨 잡기

> 전체 스케줄 보기에서 빈 슬롯 클릭 → "이 시간에 레슨 잡기" → 수강생 선택

```
[신규 수강생 등록]
  우리 서비스 외부(오프라인)에서 온 수강생을 코치가 직접 등록:

  입력 항목:
    - 이름 (필수)
    - 연락처 (필수, 휴대폰 번호 — 알림톡 발송용)
    - 레슨 유형: 정규레슨 / 쿠폰레슨 (필수)
    - 레슨 형태: 1:1 / 그룹 (필수)
    - 메모 (선택, 코치 전용 메모)

  처리:
    POST /api/coaches/external-students { name, phone, lessonType, lessonFormat, memo }
    → ExternalStudent 테이블에 저장 (코치 전용, 일반 회원과 분리)
    → 해당 시간에 Booking 생성 (status=CONFIRMED, isExternal=true)
    → Schedule isBlocked = true
    → 연락처로 카카오 알림톡 발송 (서비스 가입 유도 포함)

  특징:
    - 일반 회원(User)과 분리 관리 — ExternalStudent 테이블
    - 코치만 CRUD 가능
    - 외부 수강생이 서비스 가입 시 → 코치와 자동 매칭 (전화번호 기준)
    - 회차 카운트에 포함됨 (코치 통계용)
    - 결제는 서비스 외부에서 처리 (isPaid=false, paymentStatus='external')

[기존 수강생 선택]
  이미 서비스에 가입한 수강생 또는 이전에 등록한 외부 수강생:

  목적 선택 (필수):
    1. 날짜 변경 — 기존 레슨 일정을 이 시간으로 이동
       → 기존 Booking의 날짜/시간 업데이트
       → 기존 슬롯 isBlocked=false, 새 슬롯 isBlocked=true
       → 수강생에게 일정 변경 알림톡 발송
       → 회차 카운트 변동 없음

    2. 보강 — 기존 일정 유지 + 이 시간에 추가 레슨
       → 새 Booking 생성 (type=SUPPLEMENTARY)
       → 새 슬롯 isBlocked=true
       → 수강생에게 보강 레슨 알림톡 발송
       → 보강 회차 정책:
         정규레슨: 보강은 월간 회차(N/M)에 포함하지 않음 (별도 카운트)
         쿠폰레슨: 보강은 총 회차(N/M)에 포함됨 (1회 차감)
       → 보강 사유 입력 (선택): "결석 보충", "추가 연습", "코치 사정 보강" 등
```

## 14. 신고 정책

```
API: POST /api/reports [인증]
Request: { targetId, targetType(USER/REVIEW), reasonCode, reasonDetail }

중복 방지:
  동일 reporterId + 동일 targetId → 24시간 이내 재신고 차단
  → "이미 신고한 사용자입니다. 검토 중입니다" 토스트

신고 접수 시:
  1. Report 테이블 INSERT (status=PENDING)
  2. 관리자에게 알림 발송 (Notification: type=REPORT_RECEIVED)
```

### 14.2 신고 사유 코드 (어드민 CRUD)

| 코드 | 라벨 |
|------|------|
| `INAPPROPRIATE_MSG` | 부적절한 메시지 |
| `FAKE_PROFILE` | 허위 프로필 |
| `SPAM` | 스팸/광고 |
| `NO_SHOW` | 레슨 불이행 |
| `FRAUD` | 사기/금전 피해 |
| `OTHER` | 기타 (상세 입력 필수) |

### 14.3 관리자 조치

```
PUT /api/admin/reports/:id/action

조치 유형:
  WARNING    — 경고: 대상에게 경고 알림 발송 (계정 제한 없음)
  SUSPEND_7D — 일시 정지: 7일간 로그인 차단 + 알림
  BAN        — 영구 차단: User.isActive=false + 알림

조치 시:
  1. Report.status = RESOLVED, resolvedAt = NOW()
  2. 대상 User에 조치 적용
  3. 신고자에게 "신고 처리 결과" 알림 발송
  4. 대상에게 "조치 안내" 알림 발송
```

### 14.4 일시 정지 처리

```
SUSPEND_7D 조치 시:
  User.suspendedUntil = NOW() + 7일
  로그인 시 suspendedUntil > NOW() → "계정이 일시 정지되었습니다. 해제일: {날짜}" 토스트
  7일 경과 후 자동 해제 (스케줄러 또는 로그인 시 체크)
```

## 15. 어드민 스케줄 관리 정책

### 15.1 어드민 강제 상태 변경

```
PUT /api/admin/reschedules/:id/action [인증: 어드민]
{ actionType, reason, targetSlotId? }

actionType 목록:
  FORCE_PROPOSE    — 강제 제안 (slotIds 지정) → ScheduleHold 생성 + 수강생 알림
  FORCE_ACCEPT     — 강제 수락 (targetSlotId) → 일정 확정 + 홀드 해제 + 양측 알림
  FORCE_REJECT     — 강제 거절 → REJECTED + 수강생 알림 (기존 일정 유지)
  FORCE_COMPLETE   — 강제 완료 (newDate, newTime) → 일정 변경 + COMPLETED + 양측 알림
  EXTEND_HOLD      — 홀드 연장 (+12h) → ScheduleHold.expiresAt 갱신
  RELEASE_HOLD     — 홀드 즉시 해제 → ScheduleHold 삭제 + rescheduleStatus = PENDING
  CANCEL           — 요청 취소 → rescheduleStatus = null (초기화) + 양측 알림
  ROLLBACK         — 원래 일정 복구 → COMPLETED → 이전 일정으로 복원 + 양측 알림
  FORCE_REASSIGN   — 다른 날짜 재지정 (newDate, newTime) → 일정 변경 + 양측 알림

모든 어드민 조치는:
  1. reason 필수 입력
  2. RescheduleActionLog 테이블에 이력 기록 (adminId, actionType, reason, timestamp)
  3. 정책 설정의 "어드민 강제 변경 시 양측 알림" on일 때 알림톡 발송
```

### 15.2 어드민 슬롯 직접 관리

```
PUT /api/admin/schedules/:coachId/action [인증: 어드민]
{ actionType, slotIds?, reason }

actionType 목록:
  FORCE_BLOCK      — 특정 슬롯 강제 블록 (isBlocked=true)
  FORCE_UNBLOCK    — 특정 슬롯 강제 해제 (isBlocked=false)
  RELEASE_ALL_HOLDS — 해당 코치의 모든 홀드 즉시 해제
  RESET_SCHEDULE   — 스케줄 전체 초기화 (모든 슬롯 삭제) — 확인 다이얼로그 필수

모든 조치는 RescheduleActionLog에 기록
```

### 15.3 정책 설정 (어드민 변경 가능)

```
GET/PUT /api/admin/settings/schedule [인증: 어드민]

설정 항목:
  holdDurationHours: 12        // 홀드 유효 시간 — 전역 기본값 (시간)
                               // ※ 코치별 개별 설정 가능 (CoachProfile.holdDurationHours)
                               //    개별값 존재 시 전역값보다 우선 적용
  holdExpiryNotification: true // 만료 시 코치 알림 발송 여부
  maxConcurrentHolds: 1        // 수강생 동시 홀드 제한
  rescheduleDeadlineDays: 1            // 변경 요청 가능 기한 (D-N)
  rescheduleSelectableRange: 'this_month'  // 희망 일정 선택 범위 (this_month / next_month / 3_months)
  maxRescheduleRequestsPerMonth: 3     // 수강생 월 최대 변경 요청 횟수 (어드민 9-9 정책 설정에서 조정 가능)
  maxRecommendSlots: 5         // 자동 추천 최대 슬롯 수
  recommendRangeDays: 1        // 추천 범위 ±N일
  paidOnlyReschedule: true     // 결제 건만 변경 허용
  completedCardDays: 1         // 변경 완료 카드 노출 기간 (일)
  notifyOnRequest: true        // 변경 요청 시 코치 알림
  notifyOnUnavailable: true    // 불가 알림 시 수강생 알림
  notifyOnComplete: true       // 변경 완료 시 양측 알림
  notifyOnAdminAction: true    // 어드민 강제 변경 시 양측 알림

  // 결강 정책 (노쇼) — 어드민 9-9 "결강 정책" 섹션
  absencePriorNoticeDays: 2            // 사전 통보 기준일 D-N (1~7일)
  absenceMakeupRangeDays: 30           // 보강 일정 가능 범위 ±N일 (7~90일)
  absenceMakeupAutoActivateHours: 24   // 코치 보강 미등록 시 수강생 보강요청 활성 (1~72시간)
  absencePolicyMatrix: {
    studentNoshow:           { deductSession: true,  allowMakeup: false, allowRefund: false },
    studentNoticeAfterCutoff:{ deductSession: true,  allowMakeup: false, allowRefund: false },
    studentNoticeBeforeCutoff:{ deductSession: false, allowMakeup: true,  allowRefund: false },
    coachReason:             { deductSession: false, allowMakeup: true,  allowRefund: false },
    forceMajeure:            { deductSession: false, allowMakeup: true,  allowRefund: true  }
  }

설정값은 ScheduleSetting 테이블에 key-value로 저장
변경 시 즉시 반영 (캐시 무효화)
```

## 16. 어드민 제안 관리 정책

### 16.1 제안 통합 모델

```
어드민 제안 관리는 두 가지 제안 유형을 통합:
  1. 변경요청 제안 (RescheduleProposal): 코치가 수강생 변경요청에 응답하며 제안한 가용 일정
  2. 보강 제안 (MakeupProposal): 코치가 결강된 레슨에 대해 제안한 보강 일정

공통 속성:
  - 제안자: 코치
  - 수신자: 수강생
  - 원본 일정: 변경요청은 기존 일정, 보강은 결강된 일정
  - 제안 슬롯: 1개 이상의 가능 시간
  - 홀드: 12시간 임시 잠금 (코치별 설정값 우선)
  - 알림톡: 제안 시 자동 발송 (RESCHEDULE_PROPOSED / MAKEUP_PROPOSED)
  - 상태: PENDING / ACCEPTED / REJECTED / EXPIRED / ADMIN_FORCED
```

### 16.2 어드민 강제 조치

```
PUT /api/admin/proposals/:id/action [인증: 어드민]
{ actionType, reason, targetSlotId? }

actionType:
  FORCE_ACCEPT     — 강제 수락 (slot 지정) → 일정 확정 + 홀드 해제 + 양측 알림
  FORCE_REJECT     — 강제 거절 → REJECTED + 수강생/코치 알림
  EXTEND_HOLD      — 홀드 +12h 연장 → expiresAt 갱신
  RELEASE_HOLD     — 홀드 즉시 해제 → 슬롯 가용으로 전환 + 코치 알림
  CANCEL           — 제안 전체 취소 → 회수 + 수강생 알림
  RESEND_ALIMTALK  — 알림톡 재발송 (실패 시 또는 재요청 시)

모든 어드민 조치:
  1. reason 필수 입력
  2. 액션 로그 기록 (변경요청 → RescheduleActionLog, 보강 → MakeupActionLog)
  3. 정책 설정 notifyOnAdminAction = true 시 양측 알림톡 발송
```

### 16.3 제안 종결 및 이력 이동

```
종결 상태: ACCEPTED, REJECTED, EXPIRED, ADMIN_FORCED
종결 시각 +24시간 경과 시 → "전체 제안" 탭에서 자동 제거되고 "제안 이력" 탭에서만 조회 가능

CSV 내보내기 (GET /api/admin/proposals/export):
  컬럼: 유형, 코치, 수강생, 원본일정, 제안일정(파이프 구분), 제안시각, 상태, 종결시각, 처리자, 사유
```

### 16.4 통계 집계

```
GET /api/admin/proposals/stats?period= [인증: 어드민]

기간별 집계:
  totalCount         // 총 제안 수
  pendingCount       // 응답 대기
  acceptedCount      // 수락 완료
  rejectedCount      // 거절
  expiredCount       // 만료
  adminForcedCount   // 어드민 강제 처리

부가 지표:
  avgResponseHours   // 평균 응답 시간 (수락+거절 기준)
  acceptanceRate     // 수락율 (acceptedCount / (acceptedCount + rejectedCount + expiredCount))
  expiryRate         // 만료율
```
